# 我是怎么做出这个的

> 这不是一篇教程, 而是把整个 repo 打包成一个可以对别人讲的故事. 别人问你会不会在 GitHub 上协作时, 你就照这份底稿, 打开这个 repo 讲一遍你是怎么学会并走通的. 先照默认主线排练, 再按听众临场裁剪.

## 1. 一句话故事

> 我用一个 AI 导师, 从没碰过 GitHub, 几次就走通了完整的协作闭环: 建仓库, commit, branch, Pull Request, merge, 这个 repo 就是那趟学习的完整记录.

---

## 2. 默认故事主线

下面七幕是这个 repo 的主线, 照顺序讲就是一条完整的故事. 每一幕先看怎么讲, 再看它对应 repo 里的哪个产物.

### 第 1 幕: 发现值得学

**怎么讲**

> 我在每个项目和团队里都撞到 GitHub, 一直靠糊弄绕过去. 它显然是协作标准, 值得把核心闭环认真学会一次.

**对应产物**

> 见 [README-ORIGINAL-cn.md](../../README-ORIGINAL-cn.md) 讲 GitHub 协作为什么连非程序员也重要的部分.

### 第 2 幕: 让 AI 调研资料

**怎么讲**

> 我让 AI 把权威资料汇总起来: repository, commit, branch, Pull Request 到底是什么, 从 GitHub 官方文档来, 而不是拼凑零散博客.

**对应产物**

> 课程背后的学习素材, 汇在学习索引里.

### 第 3 幕: 蒸馏出专家

**怎么讲**

> 我把这些蒸馏成一个专注的导师, 它正好懂这一小块: 给新手的纯浏览器 GitHub 协作, 不碰命令行.

**对应产物**

> 带我学习的那个导师, 体现在 examples 的编排顺序里.

### 第 4 幕: 让专家设计一周的活

**怎么讲**

> 我告诉它我的水平和时间, 它排了一份短大纲: 建仓库, 用 commit 改, 开 branch, 用 Pull Request 合并, 再自测.

**对应产物**

> 见 [examples 索引](../README-cn.md), 那就是这份大纲.

### 第 5 幕: 每个点变成能练的例子

**怎么讲**

> 大纲每个点都变成一个我既能读又能在自己账号上做的例子, 一次正好加一个概念.

**对应产物**

> examples 下按编号排列的各个 mini task, 从 [01-create-repo](../01-create-repo/README-cn.md) 开始.

### 第 6 幕: AI 陪着学完

**怎么讲**

> 我在自己真实的 GitHub 账号上做每个 task, 卡住就找导师, 最后用自测题检验自己.

**对应产物**

> 见 [05-prove-i-get-it](../05-prove-i-get-it/README-cn.md), 确认学到位的那份自测.

### 第 7 幕: 投入使用

**怎么讲**

> 这条闭环虽小但真实: 我现在在真项目上就用它, 而同样的 "隔离, 审阅, 合并" 习惯, 换到命令行 git 也照样带得过去.

**对应产物**

> 完整闭环, 在 [04-merge-branch](../04-merge-branch/README-cn.md) 里端到端走了一遍.

## 3. 常见追问与怎么接

这些是听众可能追问的元问题 (关于这段经历和方法, 不是考具体技术; 技术细节归 quiz). 排练时准备好怎么接.

**为什么选这个技能, 而不是别的**

> 它是我能挑到的最有杠杆的基本功: 现在几乎每次协作, 无论技术不技术, 都要过 GitHub, 所以回报到处都能见到.

**这些是 AI 生成的, 你真的懂吗**

> 我能在自己账号上独立走完整条闭环, 不看教程, 能答自测题, 能讲清每一步为什么存在. AI 是加速器, 不是替我理解.

**过程中最大的坑是什么**

> 想通 branch, 以及为什么不直接在 main 上改. 把 Pull Request 看成回到 main 前那道被 review 的关卡, 是让一切都对上号的那一下.

**这套方法能复用吗**

> 这七幕是一套用 AI 快速学一个技能的可复用方法, 这个 repo 只是它的一个应用.

## 4. 按听众裁剪

对口岗位的面试官, 七幕都讲, 重点在第 5 到第 7 幕的深度. 不对口岗位的, 只讲一句话故事加第 7 幕, 或诚实建议换一个更相关的作品. 同行, 重点讲 branch 与 review 的取舍和那个坑. 非技术听众, 只讲第 1 幕和第 7 幕, 跳过机制细节.
