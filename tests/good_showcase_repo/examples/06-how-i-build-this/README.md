# How I Build This

> This is not a tutorial; it packages the whole repo into a story you can tell. When someone asks whether you can collaborate on GitHub, you follow this script, open the repo, and walk them through how you learned it and worked through it. Rehearse the default line first, then tailor to the listener on the fly.

## 1. One-line story

> I used an AI mentor to go from never having touched GitHub to walking the full collaboration loop, create, commit, branch, Pull Request, merge, in a couple of sittings, and this repo is the complete record of that run.

---

## 2. Default story arc

The seven beats below are the main line; told in order they make one complete story. For each beat, see how to tell it, then which repo artifact backs it.

### Beat 1: found it worth learning

**How to tell it**

> I kept bumping into GitHub in every project and team, and faking my way around it. It was clearly the collaboration standard, so it was worth learning the core loop properly once.

**Backing artifact**

> See [README-ORIGINAL.md](../../README-ORIGINAL.md), the part on why GitHub collaboration matters even outside programming.

### Beat 2: had AI research the material

**How to tell it**

> I asked an AI to pull together the authoritative material: what a repository, commit, branch, and Pull Request actually are, from the GitHub docs, instead of stitching together random blog posts.

**Backing artifact**

> The study material behind the course, gathered in the learning index.

### Beat 3: distilled an expert

**How to tell it**

> I distilled that into a focused mentor that knew this exact slice: browser-only GitHub collaboration for a beginner, no command line.

**Backing artifact**

> The mentor that guided the learning, reflected in how the examples are sequenced.

### Beat 4: had the expert design a week of work

**How to tell it**

> I told it my level and my time, and it laid out a short syllabus: create a repo, edit with commits, branch, merge via Pull Request, then prove-it.

**Backing artifact**

> See [the examples index](../README.md), which is that syllabus.

### Beat 5: turned each point into a hands-on example

**How to tell it**

> Each syllabus point became one example I could both read and do on my own account, adding exactly one concept at a time.

**Backing artifact**

> The numbered mini tasks under examples, from [01-create-repo](../01-create-repo/README.md) onward.

### Beat 6: learned it with AI alongside

**How to tell it**

> I did each task on my real GitHub account with the mentor on call when I got stuck, and checked myself with the self-test at the end.

**Backing artifact**

> See [05-prove-i-get-it](../05-prove-i-get-it/README.md), the self-test that confirms it stuck.

### Beat 7: put it to work

**How to tell it**

> The loop is small but real: I now use it on actual projects, and the same isolate-review-merge habit carries straight over when I move to the git command line.

**Backing artifact**

> The full loop, walked end to end across [04-merge-branch](../04-merge-branch/README.md).

## 3. Common follow-ups

These are meta questions about the journey and method, not tech trivia (that belongs to the quiz). Have answers ready.

**Why this skill and not something else**

> It is the most leveraged basic I could pick: almost every collaboration, technical or not, runs through GitHub now, so the payoff shows up everywhere.

**This was AI-generated, do you actually understand it**

> I can walk the whole loop solo on my own account without the tutorials, answer the self-test, and explain why each step exists. The AI was an accelerator, not a substitute for understanding.

**What was the biggest snag**

> Wrapping my head around branches and why you do not just edit main directly. Seeing a Pull Request as the reviewed gate back into main was the click that made it all fit.

**Does the method generalize**

> The seven beats are a reusable way to learn a skill fast with AI; this repo is just one application of it.

---

## 4. Tailoring to the audience

For a related-role interviewer, tell all seven beats, weighting beats 5 to 7 for depth. For an unrelated role, give the one-line story plus beat 7, or honestly suggest a more relevant piece. For a peer engineer, focus on the branch-and-review reasoning and the snag. For a non-technical listener, tell only beats 1 and 7 and skip the mechanics.
