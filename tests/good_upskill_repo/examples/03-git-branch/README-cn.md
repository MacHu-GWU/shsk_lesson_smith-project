# 用 Branch 隔离你的改动

> 教你在 GitHub 网页上开一个 branch, 在上面自由改文件, 而主干 (main) 一点都不受影响.

## 1. 概览

到目前为止, 你在 01 建好了 repo, 在 02 学会了直接改文件. 但你可能已经隐隐不安: 每次 commit 都直接落在主干上, 那要是改到一半, 想法又变了, 或者代码根本没跑通呢? 主干就被你的半成品污染了.

branch (分支) 就是解决这个问题的. 它让你另开一条线, 在上面随便折腾, 主干始终保持干净可用. 这一课全程在浏览器里完成, 不需要命令行.

---

## 2. 学习目标

先说为什么. 真实项目里, 主干 (通常叫 main) 代表 "随时可以拿出去用的那一版". 团队里别人会基于它继续干活, 自动化会基于它做部署. 如果你把没写完, 没测过的东西直接堆到 main 上, 相当于把工地垃圾丢进了别人的客厅.

branch 给了你一块隔离的试验田: 你的改动先待在自己的分支里, 成熟了再合回主干. 这就是所有团队协作的起点 -- 没有 branch, 多人在同一个 repo 上干活会立刻互相踩脚.

学完这个 Task, 你将能够:

1. 说清 branch 是什么, 以及默认分支 (main) 为什么特殊.
2. 在 GitHub 网页上创建一个新 branch.
3. 在新 branch 上编辑文件并提交, 同时确认 main 没有被改动.

---

## 3. 前置知识

- 完成了 01: 拥有一个自己的 repo.
- 完成了 02: 会在网页上创建, 编辑文件并写 commit message.
- 一个浏览器和你的 GitHub 账号, 不需要安装任何东西.

---

## 4. 你将学到什么

学完你会亲手拥有两条线: 一条是原封不动的 main, 一条是你新开的 branch. 你会看到同一个文件在两条线上是两个版本, 而且互不干扰. 这个 "同一个 repo, 多个平行版本" 的画面, 是理解后面 Pull Request 的关键.

---

## 5. Branch 到底是什么

想象你在写一份重要的文档. 你不确定新的改法好不好, 于是你把文件另存一个副本, 叫 "试验版", 在副本上大刀阔斧地改. 原版一直放在那儿没动. 如果试验版改得好, 你就用它替换原版; 如果改砸了, 直接删掉副本, 原版毫发无伤.

branch 就是这个 "另存一个副本" 的动作, 只不过它更聪明:

- 它不是笨拙地复制整个 repo, 而是从某个 commit 处岔出一条新的记录线.
- 你在 branch 上的每一次 commit 都只累积在这条线上, 不会碰到别的 branch.
- 随时可以对比两条线的差异, 也随时可以把一条线的成果合并进另一条.

一句话: branch 是一条独立演进的 commit 序列, 让你在不惊动主干的前提下往前推进工作.

---

## 6. 默认分支 (main) 为什么特殊

任何 repo 建出来时, GitHub 都会给你一条默认分支, 现在的约定名字是 main (老项目里可能叫 master). 它特殊在:

- 别人打开你的 repo, 首页默认展示的就是 main 上的内容.
- 它被当作 "正式版" -- 团队和自动化都默认信任 main 上的东西是可用的.
- 后面你会看到, 其它 branch 的改动最终都是奔着 "合回 main" 去的.

正因为 main 这么重要, 我们才不希望在它上面直接做没把握的改动. 规矩很简单: main 保持稳定, 试验去 branch 上做.

---

## 7. 在网页上创建一个 Branch

GitHub 网页端有个 branch switcher (分支切换器), 就是你 repo 文件列表左上方那个写着 main 的下拉按钮. 创建 branch 就在这里:

1. 打开你的 repo, 停在 Code 这个默认标签页.
2. 点击左上方那个显示 main 的下拉按钮, 它会弹出一个搜索框.
3. 在框里输入你想要的新分支名, 例如 add-tagline.
4. 下方会出现一行 Create branch add-tagline from main, 点它.
5. 页面刷新后, 那个下拉按钮上显示的就从 main 变成了 add-tagline. 这说明你现在正 "站在" 新分支上.

分支名用短横线连接的小写英文, 描述这条分支要干的事, 例如 add-tagline, fix-typo, update-readme. 这是团队里通行的习惯.

---

## 8. 在 Branch 上改文件

现在你站在 add-tagline 上, 接下来的编辑和 02 里一模一样, 唯一区别是: 改动会落在这条分支上, 而不是 main.

1. 确认左上角的按钮显示的是 add-tagline, 不是 main.
2. 点开 README.md, 点击右上角的铅笔图标进入编辑.
3. 加一行内容, 例如给项目写一句 tagline.
4. 点右上角 Commit changes, 在弹窗里写一条 commit message, 例如 Add project tagline.
5. 弹窗里注意有个选项问你提交到哪, 确认它选的是 Commit directly to the add-tagline branch, 然后确认.

提交后, 你可以再点一次那个分支下拉按钮, 在 main 和 add-tagline 之间来回切. 切到 main 时看这个文件, 它还是老样子; 切到 add-tagline, 你会看到刚加的那行. 同一个文件, 两条线, 两个版本.

---

## 9. 练习

### 练习 1: 开一条 branch 并验证 main 不受影响

**目标:** 亲眼确认在 branch 上的改动不会碰到 main.

**怎么做:**

1. 在你的 repo 里, 用 branch switcher 新建一个叫 add-tagline 的分支.
2. 站在 add-tagline 上编辑 README.md, 加一行 tagline, 提交.
3. 用分支切换器切回 main, 打开同一个 README.md.

**你会观察到:**

main 上的 README.md 里没有你刚加的那行 tagline -- 它安然如初. 切回 add-tagline, 那行又出现了.

> **关键洞见:** 改动是绑在 branch 上的. 只要你不主动合并, main 永远不会因为你在别的 branch 上折腾而改变.

---

## 10. 回顾: 我们学到了什么

- branch 是一条独立的 commit 序列, 像给文档另存一个副本来试验.
- main 是默认分支, 代表正式的, 可信任的那一版.
- branch switcher 是网页上创建和切换分支的入口.
- 在 branch 上 commit 不会影响 main, 这正是隔离的价值.

---

## 11. 导师寄语

**为什么这个练习重要:**

我带新人时, 最想让他们尽早养成的一个直觉就是: 永远不要在 main 上做没把握的事. 这不是规矩洁癖, 而是协作的地基. 当一个 repo 有十个人在改, main 就是大家共同信任的那条基线; 你的每一个半成品都待在自己的 branch 里, 别人才不会被你的中间状态绊倒.

**关键洞见:**

- 隔离不是为了藏起来, 而是为了让 "进行中" 和 "已定稿" 泾渭分明.
- 一个 branch 最好只对应一件事, 这样它以后合并和被 review 时都清清爽爽.

**下一步:**

现在你有一条改好了的 branch, 但它还孤零零地待在一边. 下一课 04 会教你用 Pull Request 把它的改动 review 之后, 正式合回 main -- 那才是协作闭环真正闭合的一步.

---

## 12. 速查

**创建 branch:** repo 的 Code 页 -> 点左上角 main 下拉 -> 输入新名字 -> 点 Create branch xxx from main.

**切换 branch:** 同一个下拉按钮里选另一条分支.

**在 branch 上提交:** 编辑文件 -> Commit changes -> 确认选的是 Commit directly to the xxx branch.

**关键约定:**

- `main`: 默认分支, 代表正式版, 保持稳定.
- 分支名: 小写加短横线, 描述这条分支要做的事, 例如 `add-tagline`.
