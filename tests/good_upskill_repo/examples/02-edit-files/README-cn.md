# 在 GitHub 上创建与编辑文件

> 用 GitHub 网页版的编辑器新建和修改文件, 理解每次保存背后的 commit, 并学会读懂 commit history.

## 1. 概览

上一个 Task 里你建好了一个 repository. 现在它基本是空的: 只有一个 README. 一个空仓库是没有意义的, 项目的价值在于文件, 而文件是一点点写出来, 一次次改出来的.

好消息是: 这些你全都能在浏览器里完成, 不用装 git, 不用碰命令行. GitHub 自带一个网页编辑器 (web editor), 新建文件, 改文件, 保存, 全在网页上点几下就行. 而每一次保存, 都会变成项目历史里一个可追溯的节点, 这个节点叫 commit.

这个 Task 教你把 "改文件" 和 "记录改动" 这两件事连起来, 建立整个 GitHub 协作里最核心的一个习惯.

---

## 2. 学习目标

先说为什么. 你在本地用 Word 写文档时, 按 Ctrl+S 保存, 文件就被覆盖了, 上一个版本消失了. 如果明天你后悔了, 想找回今天早上那一版, 基本没办法. 这就是普通保存的问题: 它只留下 "现在", 抹掉了 "过去".

GitHub 的保存不一样. 每次保存都叫一次 commit, 它不覆盖历史, 而是往历史里追加一个新节点, 同时留下一句话说明 "这次我改了什么". 于是你的项目就有了一条完整的时间线: 任何一个历史版本都能翻回去看, 任何一次改动都知道是谁, 在什么时候, 为了什么而做的. 这就是为什么工程师宁愿多点几下, 也要用 GitHub 而不是网盘来管代码.

学完这个 Task, 你将能够:

1. 用 web editor 在浏览器里新建一个文件并保存.
2. 编辑一个已有文件, 理解每次保存就是一次 commit.
3. 写出一条清晰的 commit message, 说清这次改动的意图.
4. 打开 commit history, 读懂项目走过的每一步.

## 3. 前置知识

- 一个能登录的 GitHub 账号和一个浏览器.
- 知道 commit 大致是 "一次被记录下来的改动" (Task 01 已经建立这个概念).

## 4. 你将学到什么

做完之后, 你的 recipe-box 里会多出几个文件, 并且它的 Commits 页面上会排着一串你亲手留下的 commit. 你会第一次直观看到: 一个项目不是一个静止的文件夹, 而是一条由 commit 串起来的时间线.

---

## 5. web editor: 浏览器就是你的编辑器

很多新手以为改 GitHub 上的文件必须先把仓库下载到电脑. 其实不用. GitHub 内置了一个网页编辑器, 直接在浏览器里就能写.

心智模型: 把 web editor 想成一个 "长在仓库里的记事本". 你打开它, 写字, 保存, 东西就直接进了仓库, 中间不经过你的电脑硬盘.

它适合做什么:

- 改文档, 改配置, 写 README 这类纯文本的小改动.
- 快速修个错别字, 补一行说明.

它不适合做什么:

- 一次改几十个文件的大重构 (那种以后你会用本地 git, 不在这门课范围).

对现在的你来说, web editor 已经足够把整条协作闭环走完, 这正是它的价值: 让你零安装就能上手真实的工作流.

---

## 6. 新建一个文件

我们给 recipe-box 加一个食谱文件. 假设你叫 Maria Garcia, 想放进你家的番茄意面做法.

怎么做:

1. 打开你的 recipe-box 主页, 点文件列表上方的 Add file 按钮, 选 Create new file.
2. 页面顶部有一个文件名输入框, 填 pancakes.md. 注意: 文件名里如果打一个斜杠 /, GitHub 会自动帮你建一层文件夹, 例如 recipes/pancakes.md 会把文件放进 recipes 文件夹.
3. 在下方的大编辑区里写内容, 比如几行 Markdown 格式的食材和步骤.
4. 内容写好后, 看向页面右上角的 Commit changes 按钮, 先别急着点, 下一节专门讲那个弹窗.

**你会观察到:** 页面右上角始终有一个 Commit changes 按钮. GitHub 在提醒你: 光写还不算数, 得 commit 一下, 这次改动才真正落进仓库.

---

## 7. 编辑一个已有文件

改一个已经存在的文件, 入口略有不同.

怎么做:

1. 在仓库里点开你想改的文件, 比如 README.md, 进入它的查看页面.
2. 找到文件内容右上方的铅笔图标 (悬停会显示 Edit this file), 点它进入编辑状态.
3. 直接在文本里改, 比如给 README 补一句项目介绍.
4. 改完同样点右上角的 Commit changes.

新建和编辑, 界面稍有差别, 但本质是同一件事: 你改动了仓库的内容, 然后用一次 commit 把这个改动记录下来. 记住这个统一的心智模型, 你就不会被界面细节绕晕.

---

## 8. commit message: 给未来的人留一句话

点 Commit changes 后会弹出一个小窗, 里面有一个标题框, 默认写着类似 Update README.md 的占位文字. 这一句, 就是 commit message, 是这个 Task 最该被认真对待的地方.

为什么它重要? 想象半年后, John Smith 接手你的项目, 翻到某次改动, 只看见一句 Update README.md, 他完全不知道你当时改了什么, 为什么改. 反过来, 如果那句话是 Add setup steps for new contributors, 他一眼就懂. commit message 是你写给未来的自己和队友的便条, 是项目里最廉价也最值钱的文档.

一条好的 commit message:

- 用一句话说清 "这次改动做了什么", 而不是 "改了文件".
- 像下命令一样用动词开头: Add, Fix, Update, Remove.
- 反面例子: update, fix, asdf, 改了一下. 半年后没人看得懂.
- 正面例子: Add tomato pasta recipe, Fix typo in README title.

弹窗里通常还有一个更大的描述框 (可选), 小改动留空即可, 复杂改动可以在这里多写几句展开. 填好标题, 点绿色的 Commit changes 确认, 这次改动就正式进仓库了.

---

## 9. commit history: 读懂项目的时间线

commit 一次次累积, 就成了 commit history, 也就是项目的完整时间线. 学会读它, 你才算真正理解 GitHub 在替你记什么.

怎么看:

1. 回到仓库主页, 在文件列表上方找到一个带时钟图标, 写着 Commits 的链接 (通常显示提交次数, 例如 5 commits), 点进去.
2. 你会看到一个从新到旧排列的列表, 每一行是一次 commit: 左边是你写的 commit message, 还有作者头像, 以及 "多久之前" 的时间.
3. 点任意一次 commit, 能看到那一次具体改了哪些行: 绿色是新增, 红色是删除. 这个逐行对比叫 diff.

类比: commit history 就像游戏的存档列表. 每个 commit 是一个存档点, message 是你给存档起的名字, 而 diff 是打开存档看到的 "这一档和上一档差在哪". 存档名字起得好, 将来读档才找得到.

**你会观察到:** 你写 commit message 时越用心, 这张时间线读起来就越清楚. 这是一个即时的正反馈: 好习惯当场就能看到回报.

## 10. 练习

### 练习 1: 新建一个食谱文件

**目标:** 用 web editor 给 recipe-box 新建一个文件, 并留下一条像样的 commit.

**怎么做:**

1. 在 recipe-box 里 Add file, Create new file, 命名为 recipes/pancakes.md.
2. 写几行 Markdown: 一个标题, 一份食材列表, 几步做法.
3. commit message 填 Add pancake recipe, 点 Commit changes.

**你会观察到:** 仓库主页多了 recipes 文件夹, 里面躺着 pancakes.md. 你刚刚只用浏览器就往项目里加了内容.

> **关键洞见:** 文件名里的斜杠会自动创建文件夹, 这是 web editor 的一个隐藏但常用的技巧.

### 练习 2: 改文件并读历史

**目标:** 编辑一个已有文件, 然后在 commit history 里找回这次改动.

**怎么做:**

1. 打开 README.md, 点铅笔图标, 补一句话介绍你的 recipe-box.
2. commit message 填 Update README with project intro, 提交.
3. 进 Commits 页面, 找到刚才这次 commit, 点开看它的 diff.

**你会观察到:** diff 里你新增的那句话是绿色高亮的. 项目历史精确到了行.

> **关键洞见:** 每一次保存都被永久记录, 你随时能回到任何一个历史节点, 这是普通网盘给不了的安全感.

---

## 11. 回顾: 我们学到了什么

- web editor 让你零安装, 在浏览器里就能新建和编辑仓库文件.
- 新建 (Add file) 和编辑 (铅笔图标) 入口不同, 但本质都是 "改动 + 一次 commit".
- commit message 是写给未来的便条: 用动词开头, 一句话说清做了什么.
- commit history 是项目的时间线, 每个 commit 可点开看逐行 diff.

---

## 12. 导师寄语

**为什么这个练习重要:**

我带过的新人里, 区分 "会用 git" 和 "只是能提交代码" 的, 往往就是 commit message 这一句话. 技术操作谁都能学会, 但愿不愿意为半年后的队友多花十秒钟写清楚一句话, 体现的是一个工程师的专业素养. 你现在养成的这个习惯, 会跟着你整个职业生涯.

**关键洞见:**

- 代码是写给人看的, commit message 也是. 二者都在替未来的你省时间.
- 频繁, 小步, 语义清晰的 commit, 远胜于攒一大堆改动一次性提交.

**下一步:**

在真实项目里, 试着让每一次 commit 只做一件事. 加功能就是加功能, 修错字就是修错字, 别混在一起. 这样你的 commit history 会像一本读得懂的日记, 而不是一团乱麻.

---

## 13. 速查

**新建文件:** 仓库主页 -> Add file -> Create new file -> 填文件名 (斜杠自动建文件夹) -> 写内容 -> Commit changes.

**编辑文件:** 点开文件 -> 铅笔图标 (Edit this file) -> 改内容 -> Commit changes.

**看历史:** 仓库主页 -> Commits (时钟图标) -> 点某次 commit 看 diff.

**好的 commit message:** 动词开头, 一句话讲清意图. 例如 Add tomato pasta recipe.
